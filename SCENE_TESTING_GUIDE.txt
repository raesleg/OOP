PART A: OOP VIOLATION LOG  (Pre-Refactor Audit)
================================================================================

The following violations were identified in the codebase before fixes.
Each entry cites the class, the rule violated, and why it matters.

VIOLATION 1: Scenes Hardcode Raw Key Codes
───────────────────────────────────────────
  Where:   GameScene.handleInput()  — `Input.Keys.ESCAPE`
           PauseScene.handleInput() — `Input.Keys.ESCAPE`, `.W`, `.S`, `.ENTER`
           StartScene.handleInput() — `Input.Keys.ENTER`
  Rule:    Encapsulation / Single Responsibility Principle (SRP)
  Problem: Each Scene imported `com.badlogic.gdx.Input` and hard-keyed specific
           key codes.  If the user wants to rebind "Pause" from ESC to P, every
           Scene that checks for ESC must be found and edited.  Scenes should
           ask for *intent* ("is pause requested?"), not raw data.
  Fix:     IOManager now exposes only semantic methods (isPauseRequested, etc.).
           Scenes no longer import Input.

VIOLATION 2: IOManager Instantiated Per Scene (No Single Source of Truth)
─────────────────────────────────────────────────────────────────────────
  Where:   GameScene.show()  — `ioManager = new IOManager();`
           PauseScene.show() — `ioManager = new IOManager();`
           StartScene.show() — `ioManager = new IOManager();`
  Rule:    Single Source of Truth / Dependency Injection
  Problem: Every Scene creates its own IOManager with its own default key
           bindings. If a future "Options" scene rebinds a key, that change
           is local to that one IOManager instance; the GameScene would still
           use the original bindings.
  Fix:     IOManager now holds a `static` shared key-binding configuration
           (InputConfig). All instances read from the same config, so a
           rebind in any Scene applies everywhere. A future enhancement is
           to inject a single IOManager via the SceneManager constructor.

VIOLATION 3: resize() Was a No-Op in Every Scene
─────────────────────────────────────────────────
  Where:   Scene.java             — `public void resize(…) { }`
           GameScene.resize()     — empty body
           StartScene.resize()    — empty body
           PauseScene.resize()    — empty body
  Rule:    Liskov Substitution / Correct contract implementation
  Problem: The window could be freely resized but nothing updated — the
           game image stretched and distorted instead of scaling properly.
  Fix:     Scene base class now initialises a Viewport + Camera and calls
           `viewport.update(w, h, true)` in resize(). Subclasses call
           super.resize().

VIOLATION 4: PPM / World Size Coupled to Window Pixels
───────────────────────────────────────────────────────
  Where:   GameScene fields — `Gdx.graphics.getWidth() / PPM`
           GameScene.show() — `physicsWorld.createBoundsPixels(Gdx.graphics.getWidth(), …)`
  Rule:    Loose Coupling / No Magic Numbers
  Problem: World width/height were calculated from the *actual* window size at
           field-initialisation time. Resizing the window changed nothing,
           and the physics bounds didn't match the rendered viewport.
  Fix:     World dimensions are now derived from VIRTUAL_WIDTH / VIRTUAL_HEIGHT
           (1280 × 720), which is the design resolution used by the Viewport.
           Physics bounds are constant and match the camera's visible area.

VIOLATION 5: ControlState.UserControlled Hardcoded Key Constants
────────────────────────────────────────────────────────────────
  Where:   ControlState.java — `io.isKeyDown(Input.Keys.A)`, etc.
  Rule:    Encapsulation
  Problem: The movement controller bypassed IOManager's semantic layer and
           queried raw key codes directly.
  Fix:     UserControlled now calls `io.isLeftHeld()`, `io.isUpHeld()`, etc.

VIOLATION 6: PauseScene Rendered in World Space, Not Screen Space (Feb 13)
──────────────────────────────────────────────────────────────────────────
  Where:   PauseScene.render() — used virtual coordinates for overlay/UI
           Hardcoded box dimensions (500×320 px)
           Fixed font scales (4f, 2.5f, 1.5f) without adaptation
  Rule:    Separation of Concerns / Adaptation / Encapsulation
  Problems:
    • Used VIRTUAL_WIDTH/HEIGHT for overlay = clipped by FitViewport letterboxing
    • Fixed box dimensions didn't adapt to screen size → UI broken on different resolutions
    • Fixed font scales → unreadable at small resolutions, tiny at large ones
    • Used camera.combined projection → overlay only covered game world, not full screen
    • No support for window resizing updates to UI dimensions
  Impact:  Pause menu didn't cover full screen, was unreadable on non-1280×720 resolutions
  Fix:     See VIOLATION 6 FIX below.

PART B: MODIFICATIONS & FIXES  (February 13, 2026)
================================================================================

FIX 1: PauseScene Screen-Space Rendering (Separation of Concerns)
──────────────────────────────────────────────────────────────────
  Files Modified: PauseScene.java

  Changes Applied:
    1) Added `import com.badlogic.gdx.math.Matrix4;`
    2) Added protected field:
       - `private Matrix4 screenProjection;`
       to cache screen-to-clip-space projection matrix
    3) In show():
       - Initialize screenProjection = new Matrix4().setToOrtho2D(0, 0,
         Gdx.graphics.getWidth(), Gdx.graphics.getHeight())
    4) In resize(int width, int height):
       - Update screenProjection.setToOrtho2D(0, 0, width, height)
       to match new window size
    5) In render(SpriteBatch batch):
       - Get int screenWidth = Gdx.graphics.getWidth()
       - Get int screenHeight = Gdx.graphics.getHeight()
       - Use screenProjection (not camera.combined) as projection matrix
       - Draw overlay with shapeRenderer.rect(0, 0, screenWidth, screenHeight)
         to cover entire GL viewport (not just virtual world)
       - Box dimensions proportional to screen:
         boxWidth = screenWidth * 0.5f;
         boxHeight = screenHeight * 0.5f;
       - Font scales adaptive:
         float scaleFactor = screenHeight / VIRTUAL_HEIGHT;
         font.getData().setScale(4f * scaleFactor)  // Title scales with screen
         font.getData().setScale(2.5f * scaleFactor) // Options scale with screen
         font.getData().setScale(1.5f * scaleFactor) // Instructions scale
       - All positioning uses screen coordinates, multiplied by scaleFactor

  OOP Principles Upheld:
    • Separation of Concerns: UI/overlay rendering isolated from world rendering
    • Encapsulation: screenProjection matrix cached, updated on resize
    • Single Responsibility: PauseScene manages its own screen-space rendering
    • DRY (Don't Repeat Yourself): scaleFactor computed once, reused for all text

FIX 2: SceneManager Calls resize() on New Scenes (Initialization Order)
────────────────────────────────────────────────────────────────────────
  Files Modified: SceneManager.java

  Changes Applied:
    1) In push(Scene scene):
       - After scene.show(), call scene.resize(Gdx.graphics.getWidth(), Gdx.graphics.getHeight())
         to initialize viewport with actual screen dimensions
    2) In set(Scene scene):
       - After scene.show(), call scene.resize(Gdx.graphics.getWidth(), Gdx.graphics.getHeight())
         for same reason

  Why This Matters:
    • Previously, a new scene's viewport was (0, 0) until user manually resized window
    • Now viewport is initialized with correct screen dimensions immediately
    • Scenes render correctly from the moment they are shown

FIX 3: Font Scaling Improvements (Readability)
───────────────────────────────────────────────
  Files Modified: StartScene.java, GameScene.java, PauseScene.java

  Changes Applied:
    • StartScene title scale: 3f → 5f
    • StartScene prompt scale: 1.5f → 2.5f
    • GameScene HUD scale: 1f (default) → 2f
    • PauseScene title scale: 2.5f → 4f
    • PauseScene options scale: 1.5f → 2.5f
    • PauseScene instructions scale: 1f → 1.5f
    + PauseScene: scales further multiplied by (screenHeight / VIRTUAL_HEIGHT)

  Result: Text is now visible and readable at any resolution

PART B: IMPLEMENTATION GAPS  (Pre-Refactor)
================================================================================

GAP 1: GameScene Not Fully Utilising IOManager
───────────────────────────────────────────────
  The GameScene only used IOManager for ESC detection.  Movement was entirely
  delegated to ControlState.UserControlled, which itself bypassed IOManager's
  semantic API.  After refactor, both systems route through the same semantic
  methods and the same shared key configuration.

GAP 2: No Viewport in Any Scene
────────────────────────────────
  No scene created a Viewport or Camera.  The SpriteBatch used the default
  screen-pixel projection, meaning font sizes changed with window size.
  After refactor, every Scene has a Viewport.  GameScene uses ExtendViewport
  (world grows when the window is wider); StartScene and PauseScene use
  FitViewport (letterboxed, UI stays pixel-identical).

GAP 3: ShapeRenderer Projection Not Synchronised
─────────────────────────────────────────────────
  In the original GameScene, shapeRenderer used `batch.getProjectionMatrix()`
  — but batch itself was never given the camera projection.  After refactor,
  both batch and shapeRenderer receive `camera.combined`, so friction zones
  and entity sprites align exactly.

GAP 4: PauseScene Could Not Cover Full Screen (Feb 13)
───────────────────────────────────────────────────────
  Pause overlay was clipped by FitViewport letterbox bars. It only covered
  the virtual world rectangle (1280×720), leaving black bars visible when
  the actual screen was larger. Now it uses screen-space rendering to cover
  the entire window regardless of viewport letterboxing.

PART C: HOW THE NEW SYSTEMS WORK (Usage Guide)
================================================================================

─── C.1  How Key Bindings Work ─────────────────────────────────────────────────

Key bindings live in `IOManager.InputConfig` (a static inner class).
Every IOManager instance shares the same InputConfig by default.

    ┌──────────────────────────────────────────────────┐
    │  IOManager.InputConfig   (shared, static)        │
    │  ────────────────────────────────────             │
    │  "PAUSE"   → ESC                                 │
    │  "CONFIRM" → ENTER  (alt: NUMPAD_ENTER)          │
    │  "UP"      → W      (alt: UP arrow)              │
    │  "DOWN"    → S      (alt: DOWN arrow)             │
    │  "LEFT"    → A      (alt: LEFT arrow)             │
    │  "RIGHT"   → D      (alt: RIGHT arrow)            │
    │  "ACTION"  → SPACE                               │
    └──────────────────────────────────────────────────┘

To query:
    ioManager.isPauseRequested()   // true the frame ESC is first pressed
    ioManager.isConfirmRequested() // true the frame ENTER is first pressed
    ioManager.isUpHeld()           // true every frame W or UP is held

─── C.2  How to Add a New Keybind ──────────────────────────────────────────────

Step 1: Add a constant in IOManager:
    public static final String SPRINT = "SPRINT";

Step 2: Register the default binding in InputConfig.defaults():
    primary.put(SPRINT, Input.Keys.SHIFT_LEFT);

Step 3: Add semantic methods in IOManager:
    public boolean isSprintHeld() { return isActionHeld(SPRINT); }

Step 4: Use it in your Scene:
    if (ioManager.isSprintHeld()) { speed *= 2; }

That's it.  No Scene ever imports `com.badlogic.gdx.Input`.

─── C.3  How to Rebind a Key at Runtime ────────────────────────────────────────

    // In an OptionsScene (future):
    IOManager.globalConfig().rebind(IOManager.PAUSE, Input.Keys.P);

This change applies immediately to every IOManager instance in every Scene.

─── C.4  Viewport System ──────────────────────────────────────────────────────

The base Scene class handles viewport setup and `resize()`:

  • Scene constructor creates the camera.
  • Subclasses override `createViewport()` to choose their viewport type.
  • `resize()` in Scene calls `viewport.update(w, h, true)`.

GameScene returns an ExtendViewport (sees more world when the window grows).
StartScene / PauseScene / Overlay scenes return a FitViewport (letterboxed, stable UI).

Every render() method must call:
    viewport.apply();
    camera.update();
    batch.setProjectionMatrix(camera.combined);
    // And for ShapeRenderer:
    shapeRenderer.setProjectionMatrix(camera.combined);

For screen-space overlays (like PauseScene), use:
    Matrix4 screenProjection = new Matrix4().setToOrtho2D(0, 0, screenWidth, screenHeight);
    batch.setProjectionMatrix(screenProjection);
    // Now rendering is in screen pixels (0, 0) to (screenWidth, screenHeight)

─── C.5  Virtual Resolution ────────────────────────────────────────────────────

All scenes share a design resolution:
    Scene.VIRTUAL_WIDTH  = 1280
    Scene.VIRTUAL_HEIGHT = 720

Use these constants — never use Gdx.graphics.getWidth/Height for layout
in world-space rendering. However, DO use Gdx.graphics.getWidth/Height
for screen-space overlay calculations and font scaling.

PART D: TESTING GUIDE
================================================================================

HOW TO RUN THE GAME
───────────────────
    .\gradlew lwjgl3:run

TEST 1: Basic Navigation
─────────────────────────
  1. Launch → see Start Menu (text clearly readable)
  2. Press ENTER → Game starts
  3. Move with WASD → Player moves
  4. Press ESC → Pause Menu appears (covers entire window)
  5. Verify menu is readable and no black bars visible  ✓
  6. Press ESC again → Resume game
  7. Verify player position preserved  ✓

TEST 2: Pause Menu Navigation
──────────────────────────────
  1. In Game, press ESC → Pause Menu
  2. Verify menu box covers full window (no letterbox bars showing)  ✓
  3. Press S (down) → "Exit to Main Menu" highlighted
  4. Press W (up) → "Resume" highlighted
  5. Press ENTER → Resume game

TEST 3: Exit to Main Menu
──────────────────────────
  1. In Game, press ESC → Pause Menu
  2. Navigate to "Exit to Main Menu", press ENTER
  3. Start Menu shown → press ENTER → new game
  4. Verify game time reset to 0  ✓

TEST 4: Window Resize (NEW - Feb 13)
────────────────────────────────────
  1. Start game (GameScene)
  2. During gameplay, drag window corner to resize
  3. Verify:
     - GameScene: world grows/shrinks smoothly, no stretching (ExtendViewport) ✓
     - Text remains readable ✓
  4. Press ESC → Pause Menu
  5. Verify pause overlay covers ENTIRE window (including letterbox area) ✓
  6. Resize window further
  7. Pause menu scales and re-centers properly ✓
  8. Text in pause menu scales with screen height ✓

TEST 5: Resolution Independence
────────────────────────────────
  1. Close the game
  2. Resize the window to a small size (e.g., 640×360)
  3. Run the game
  4. Start Menu text should be readable  ✓
  5. Press ENTER
  6. Pause menu (ESC) should a) cover full window, b) be readable  ✓
  7. Resize window to large (e.g., 1920×1080)
  8. Resume game
  9. Everything still readable and properly scaled  ✓

TEST 6: Game State Preservation
────────────────────────────────
  1. Start game, move player to a corner, wait 10+ seconds
  2. ESC → Pause → ESC → Resume
  3. Player still in corner, timer continues  ✓
