OOP VIOLATIONS REPORT
=====================
What was changed, why it was a violation, and how it was fixed.


1. ENCAPSULATION — MotionTuning exposed public fields
------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/MotionTuning.java

What was wrong:
  All four fields (maxSpeed, maxForce, lateralGrip, linearDamping) were
  declared as "public final" instead of "private final" with getters.

Why it violates OOP:
  Encapsulation requires that an object's internal data is hidden behind
  an access-controlled interface. Public fields let any external class
  couple directly to the internal representation, making future changes
  (validation, derived values, unit conversion) impossible without
  modifying every consumer.

How it was fixed:
  Changed all four fields to "private final" and added getter methods:
  getMaxSpeed(), getMaxForce(), getLateralGrip(), getLinearDamping().
  Updated FrictionMovement.java to call the getters instead of accessing
  fields directly (e.g. tuning.maxSpeed -> tuning.getMaxSpeed()).


2. ENCAPSULATION — Scene exposed protected fields
-----A---------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/scene/Scene.java

What was wrong:
  All 11 fields (sceneManager, ioManager, controls, camera, viewport,
  uiCamera, uiViewport, entityManager, movementManager, collisionManager)
  were declared "protected", giving every subclass direct field access.

Why it violates OOP:
  Protected fields expose internal representation to the entire
  inheritance hierarchy. Any subclass can bypass invariants, and changes
  to how a field is stored or initialised force modifications across all
  subclasses. This is the "fragile base class" problem.

How it was fixed:
  Changed all fields to "private". Added protected getters:
  getSceneManager(), getIOManager(), getControls(), getCamera(),
  getViewport(), getUiCamera(), getUiViewport(), getEntityManager(),
  getMovementManager(), getCollisionManager(). Added protected setters:
  setEntityManager(), setMovementManager(), setCollisionManager(),
  setControls(). Updated all three subclasses (GameScene, StartScene,
  PauseScene) to use the accessor methods instead of direct field access.


3. ENCAPSULATION / DRY — GameScene.isMoving duplicated sound state
------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameScene.java

What was wrong:
  A "private boolean isMoving" field manually shadowed whether the
  move-loop sound was playing, duplicating state already tracked inside
  SoundManager. The two could drift out of sync.

Why it violates OOP:
  Maintaining a shadow copy of another object's state breaks
  encapsulation (the scene "knows" internal implementation details of the
  sound system) and violates DRY (two sources of truth). A bug in either
  update path silently causes inconsistency.

How it was fixed:
  Removed the isMoving field. updateMoveLoop() now queries
  sound.isLooping("move") instead. Added isLooping(String) to the
  SoundDevice interface and implemented it in SoundManager via
  SoundEffect.isLooping(). stopMoveLoop() no longer sets a flag.


4. ENCAPSULATION / DRY — ExplosionParticle duplicated width/height
------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/ExplosionParticle.java

What was wrong:
  The class declared its own "private final float width" and
  "private final float height" fields, duplicating the same values
  already stored in the parent Entity class.

Why it violates OOP:
  Duplicating inherited state wastes memory, introduces the risk of
  inconsistency if one copy is updated but not the other, and ignores
  the encapsulated interface the superclass already provides via
  getW() and getH().

How it was fixed:
  Removed the duplicate fields. The draw() method now calls the
  inherited getW() and getH() accessors.


5. DEPENDENCY INVERSION PRINCIPLE — GameMaster depended on concrete classes
---------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/GameMaster.java

What was wrong:
  GameMaster (a high-level engine class) directly imported and
  instantiated Keyboard and StartScene (low-level demo classes) inside
  its create() method.

Why it violates OOP:
  The Dependency Inversion Principle states that high-level modules
  should depend on abstractions, not concrete implementations.
  Hardcoding demo-specific classes in the engine makes the engine
  non-reusable and tightly coupled to a single game.

How it was fixed:
  GameMaster now accepts a Scene and an InputDevice through its
  constructor. The composition root in Lwjgl3Launcher.java now passes
  "new StartScene()" and "new Keyboard()", keeping all concrete wiring
  at the application boundary.


6. POLYMORPHISM / OPEN-CLOSED PRINCIPLE — MovableEntity used instanceof
-----------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/movement/MovableEntity.java

What was wrong:
  The constructor contained "this.aiControlled = controls instanceof
  AIControlled" and stored a redundant boolean flag. isAIControlled()
  returned that flag.

Why it violates OOP:
  instanceof checks defeat polymorphism. Adding a new non-player
  ControlSource (e.g. network-controlled) would require modifying
  MovableEntity. The Open-Closed Principle says classes should be open
  for extension but closed for modification. The separate boolean also
  duplicates state.

How it was fixed:
  Removed the aiControlled field and the instanceof check. Added a
  "default boolean isPlayerControlled()" method to the ControlSource
  interface (returns true). AIControlled overrides it to return false.
  MovableEntity.isAIControlled() now delegates to
  !controls.isPlayerControlled().


7. LISKOV SUBSTITUTION PRINCIPLE — Shape forced to implement meaningless methods
--------------------------------------------------------------------------------
Files: core/src/main/java/io/github/raesleg/engine/entity/Entity.java
       core/src/main/java/io/github/raesleg/engine/entity/Shape.java

What was wrong:
  Entity declared "abstract void draw(SpriteBatch batch)" and
  "abstract void dispose()". Shape (which renders via ShapeRenderer,
  not SpriteBatch) was forced to provide empty no-op implementations
  of both methods.

Why it violates OOP:
  The Liskov Substitution Principle requires that subclasses can
  meaningfully honour their parent's contract. Forcing Shape to
  implement a SpriteBatch-based draw method it can never use means the
  abstraction is wrong. The empty overrides are dead code that misleads
  readers.

How it was fixed:
  Changed draw(SpriteBatch) and dispose() from abstract to concrete
  methods with empty default bodies in Entity. Removed the empty
  overrides from Shape. Subclasses that need sprite-based drawing
  (like MovableEntity) still override them.


8. LAW OF DEMETER — GameScene reached through bucket to query velocity
----------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameScene.java

What was wrong:
  Two places in GameScene called
  "bucket.getPhysicsBody().getVelocity().len()" — a three-level chain
  that couples the scene to the internal structure of MovableEntity's
  physics implementation.

Why it violates OOP:
  The Law of Demeter ("don't talk to strangers") says an object should
  only call methods on its direct collaborators, not reach deep into
  their internals. If the physics body representation changes, GameScene
  would break even though it has no direct relationship with PhysicsBody.

How it was fixed:
  Both call sites now use bucket.isMoving(), a method already exposed
  by MovableEntity that encapsulates the velocity check.


9. DRY — GameCollisionHandler duplicated A/B instanceof checks
--------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameCollisionHandler.java

What was wrong:
  Both beginContact and endContact contained four mirrored
  "if (a instanceof X && b instanceof Y) ... else if (a instanceof Y
  && b instanceof X)" blocks — identical logic with A and B swapped.

Why it violates OOP:
  DRY states every piece of knowledge should have a single, unambiguous
  representation. Duplicating the swap logic means a change to collision
  matching must be made in two mirrored branches, doubling the chance of
  error.

How it was fixed:
  Added a private record ZoneCollision(MovableEntity, MotionZone) and a
  helper method extractZoneCollision(Entity, Entity) that normalises
  the pair regardless of order. Each contact method now calls the helper
  once and acts on the result, eliminating the mirrored branches.


10. ENCAPSULATION — SoundDevice interface lacked loop-state query
-----------------------------------------------------------------
Files: core/src/main/java/io/github/raesleg/engine/io/SoundDevice.java
       core/src/main/java/io/github/raesleg/engine/sound/SoundManager.java
       core/src/main/java/io/github/raesleg/engine/sound/SoundEffect.java

What was wrong:
  There was no way to ask the sound system whether a sound was currently
  looping. This forced GameScene to maintain its own shadow isMoving
  flag (see violation #3).

Why it violates OOP:
  When an object owns a piece of state but doesn't expose it through
  its interface, consumers are forced to duplicate that state externally,
  breaking encapsulation.

How it was fixed:
  Added "boolean isLooping(String name)" to SoundDevice. Implemented it
  in SoundManager by delegating to a new SoundEffect.isLooping() method
  that checks whether a loop ID is active.

11. OPEN-CLOSED PRINCIPLE — GameCollisionHandler used instanceof FrictionMovement
---------------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameCollisionHandler.java

What was wrong:
  Both onCollisionBegin and onCollisionEnd checked
  "zc.movable().getMovementModel() instanceof FrictionMovement fm"
  before calling onEnterZone()/onExitZone(). The collision handler was
  coupled to a specific concrete class.

Why it violates OOP:
  If a new MovementModel that also supports zone transitions were added
  (e.g. SlipperyMovement), GameCollisionHandler would need to be
  modified to add another instanceof check. The Open-Closed Principle
  says classes should be open for extension but closed for modification.

How it was fixed:
  Added default onEnterZone(PhysicsBody, Object) and
  onExitZone(PhysicsBody) methods to the MovementModel interface (as
  no-ops). FrictionMovement now overrides them with @Override. The
  collision handler calls model.onEnterZone()/onExitZone()
  polymorphically on any MovementModel, with no instanceof check.


12. LAW OF DEMETER — GameScene used getIOManager().getSound().isMuted()
-----------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameScene.java

What was wrong:
  In render(), the mute status was retrieved via a two-level method
  chain: "getIOManager().getSound().isMuted()". The SoundDevice was
  already cached in the local "sound" field.

Why it violates OOP:
  The Law of Demeter says an object should only call methods on its
  direct collaborators. The chain reaches through IOManager to
  SoundDevice, coupling render() to IOManager's internal structure,
  when a direct reference already exists.

How it was fixed:
  Replaced "getIOManager().getSound().isMuted()" with
  "sound.isMuted()" using the already-cached field.


13. DRY — Sound registration duplicated across three scenes
------------------------------------------------------------
Files: core/src/main/java/io/github/raesleg/demo/GameScene.java
       core/src/main/java/io/github/raesleg/demo/StartScene.java
       core/src/main/java/io/github/raesleg/demo/PauseScene.java
       core/src/main/java/io/github/raesleg/engine/GameMaster.java

What was wrong:
  sound.addSound("selected", "uiSelected_sound.wav") appeared in three
  scenes, and sound.addSound("menu", "uiMenu_sound.wav") in two. Each
  scene independently re-registered the same shared UI sounds.

Why it violates OOP:
  DRY states every piece of knowledge should have a single unambiguous
  representation. Duplicating sound-asset paths across scenes means a
  filename change must be made in multiple places.

How it was fixed:
  Moved the shared UI sound registrations ("menu" and "selected") to
  GameMaster.create(), where the SoundDevice is first created. Removed
  the duplicate addSound() calls from all three scenes. Scene-specific
  sounds (like "move" and "explosion") remain in their owning scene.


14. DRY — ExplosionParticle.draw() duplicated TextureObject.draw() logic
------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/ExplosionParticle.java

What was wrong:
  ExplosionParticle.draw() contained the same centering calculation
  as TextureObject.draw():
    batch.draw(getTexture(), getX()-getW()/2f, getY()-getH()/2f, ...)
  The only addition was alpha blending around it.

Why it violates OOP:
  DRY forbids copy-pasting logic that already exists in a parent class.
  If the centering formula changes in TextureObject, the copy in
  ExplosionParticle would silently become inconsistent.

How it was fixed:
  Replaced the duplicated draw call with super.draw(batch), which
  reuses the parent's centering logic. The alpha blending wrapper
  (setColor/restore) remains in the override.


15. LAW OF DEMETER — FrictionMovement reached through MovableEntity to ControlSource
-------------------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/FrictionMovement.java
      core/src/main/java/io/github/raesleg/engine/movement/MovableEntity.java

What was wrong:
  FrictionMovement.step() called "e.getControls().getX(dt)" and
  "e.getControls().getY(dt)", reaching through MovableEntity to its
  ControlSource collaborator.

Why it violates OOP:
  The Law of Demeter says a method should only talk to its direct
  parameters, not to objects obtained from them. FrictionMovement
  should not need to know that MovableEntity delegates input to a
  separate ControlSource object.

How it was fixed:
  Added getInputX(float dt) and getInputY(float dt) delegate methods
  to MovableEntity. FrictionMovement now calls e.getInputX(dt) and
  e.getInputY(dt) directly, without reaching through to ControlSource.


16. ENCAPSULATION — Scene.setSceneManager() and setIOManager() were public
--------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/scene/Scene.java

What was wrong:
  setSceneManager(SceneManager) and setIOManager(IOManager) were
  declared "public", allowing any class in any package to rewire a
  scene's manager references.

Why it violates OOP:
  Encapsulation requires restricting access to the minimum necessary.
  These methods are only called by SceneManager (same package). Making
  them public lets demo-layer code accidentally or intentionally
  overwrite the wiring, breaking the framework's guarantees.

How it was fixed:
  Changed both methods from "public" to package-private (default
  access). Only classes in io.github.raesleg.engine.scene (i.e.
  SceneManager) can call them.

17. DEPENDENCY INVERSION PRINCIPLE — MovementModel.step depended on concrete MovableEntity
-----------------------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/movement/MovementModel.java

What was wrong:
  The engine-level `MovementModel.step` method used the concrete `MovableEntity`
  type as its parameter. This couples the abstract movement behaviour to a
  specific game-layer implementation.

Why it violates OOP:
  A high-level engine abstraction should depend on an interface rather than a
  concrete class (Dependency Inversion Principle). Relying on `MovableEntity`
  prevents other movable implementations from reusing the movement models.

How it was fixed:
  Added the required accessors to the existing `IMovable` interface
  (`getPhysicsBody()`, `getInputX(float)`, `getInputY(float)`) and changed
  `MovementModel.step(MovableEntity, float)` to `MovementModel.step(IMovable, float)`.
  Updated `FrictionMovement` to accept `IMovable` and `MovableEntity` already
  implements `IMovable`, so no game-layer changes were required beyond the
  method signature update.


18. DEPENDENCY INVERSION PRINCIPLE — UI sound registration before LibGDX init
---------------------------------------------------------------------------
File: lwjgl3/src/main/java/io/github/raesleg/OOP/lwjgl3/Lwjgl3Launcher.java
      core/src/main/java/io/github/raesleg/engine/GameMaster.java

What was wrong:
  The launcher created a `SoundManager` and immediately called `addSound(...)`
  before LibGDX had been initialised. `SoundEffect` attempts to access
  `Gdx.files` during construction which is null before `Lwjgl3Application`
  starts, causing a runtime `NullPointerException` when running via Gradle.

Why it violates OOP / runtime safety:
  This is an initialization-order error: composition/root wiring must avoid
  performing framework-dependent work before the framework is ready.

How it was fixed:
  Stop registering sound assets in the launcher. The launcher now constructs
  the `SoundManager` and passes it to `GameMaster`. `GameMaster.create()` —
  which runs after LibGDX is initialised — now registers the shared UI sounds
  (`"menu"` and `"selected"`). This preserves Dependency Inversion while
  ensuring framework resources are available before asset loading.


19. LISKOV SUBSTITUTION PRINCIPLE — `Scene.handleInput` unnecessarily abstract
-----------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/engine/scene/Scene.java
      core/src/main/java/io/github/raesleg/demo/GameScene.java
      core/src/main/java/io/github/raesleg/demo/StartScene.java
      core/src/main/java/io/github/raesleg/demo/PauseScene.java

What was wrong:
  `Scene.handleInput(float)` was declared abstract, but the three concrete
  scene subclasses implemented empty no-op overrides. This meant the base
  contract forced meaningless implementations.

Why it violates OOP:
  For the Liskov Substitution Principle, a superclass should provide a
  usable contract. Making `handleInput` abstract forced subclasses to provide
  empty methods when no input handling was required.

How it was fixed:
  Converted `Scene.handleInput(float)` to a concrete no-op implementation in
  the base class. Removed the empty overrides from `GameScene`, `StartScene`,
  and `PauseScene`. Subclasses may still override when they need per-frame
  input processing.


20. SINGLE RESPONSIBILITY PRINCIPLE — Particle spawn math in collision handler
-----------------------------------------------------------------------------
File: core/src/main/java/io/github/raesleg/demo/GameCollisionHandler.java
      core/src/main/java/io/github/raesleg/demo/ExplosionParticle.java

What was wrong:
  `GameCollisionHandler.createParticleExplosion(...)` contained detailed
  visual spawning math (angles, random scatter, velocities) and instantiated
  `ExplosionParticle` objects. This mixes collision/business logic with
  presentation/visual concerns.

Why it violates OOP:
  The collision handler's responsibility is to resolve physical collisions;
  creating and tuning visual particles is a separate concern (SRP). Mixing
  them reduces clarity and reusability.

How it was fixed:
  Added a static factory method `ExplosionParticle.spawnExplosion(EntityManager, Vector2, float)`
  that encapsulates the entire visual spawn logic. `GameCollisionHandler`
  now delegates to this method and no longer contains the particle math.
